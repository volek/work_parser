# Целевой дизайн: `default_warm` + warm-индексация массивов (включая `object`)

## 1. Контекст и цель

Планируемый переход:

- базовая стратегия хранения: `default`, но только с ограниченным набором warm-полей;
- механизм из `compcom` для warm-путей используется для индексирования массивов;
- хранение массивов целиком в формате JSON исключается.

Цель дизайна:

- сохранить простую и стабильную основную модель (main datasource);
- поддержать поиск по значениям внутри массивов, включая object-поля;
- избежать взрывного роста числа колонок и uncontrolled dynamic schema.

---

## 2. Архитектура решения

Используются два datasource:

1. `default_warm_main` (основной)
   - одна запись на сообщение;
   - фиксированные process-поля;
   - фиксированный набор warm-колонок;
   - без полного flatten всех переменных;
   - без хранения массивов целиком в JSON.

2. `default_warm_arrays_indexed` (индексный слой массивов)
   - хранит только значения из массивов по whitelist-путям;
   - правила подключения путей такие же, как в `compcom` (через warm categories и paths с `[*]`);
   - записи создаются только для разрешенных warm leaf-полей;
   - object-поля поддерживаются в управляемых режимах хранения.

Итог: основной datasource остается компактным и предсказуемым, а поиск по массивам переносится в отдельный управляемый индексный слой.

---

## 3. Принцип подключения массивов

Массивы подключаются строго конфигурацией warm-путей:

- используется existing логика категорий (`tier2WarmCategories`);
- для массивов путь задается через wildcard `[*]`;
- нет пути в конфиге -> данные из массива не сохраняются.

Для каждого пути массива задается режим обработки значения:

- `leaf_scalar` - индексируются только scalar/null leaf-значения;
- `leaf_object_flatten` - object разворачивается в разрешенные leaf warm-поля;
- `raw_object_json` - object/array сохраняется целиком как canonical JSON (только если явно разрешено).

Примеры разрешенных путей:

- `epkEntity.names[*].surname`
- `epkEntity.phoneNumbers[*].phoneNumber`
- `attributes[*].inputCC.DPAN`
- `attributes[*].inputCC` + mode `leaf_object_flatten`
- `attributes[*].metadata` + mode `raw_object_json` (опционально)

Это означает режим `deny by default` для всех массивов и их полей.

---

## 4. Формат данных в `default_warm_main`

`default_warm_main` хранит:

- идентификаторы и тайм-метки процесса (`process_id`, `__time`, `state`, и т.д.);
- ограниченный набор warm-полей верхнего и структурированного уровня;
- поля с фиксированными типами и именами колонок.

`default_warm_main` не хранит:

- все переменные сообщения как отдельные динамические колонки;
- массивы целиком в JSON;
- неразрешенные warm-пути.

---

## 5. Формат данных в `default_warm_arrays_indexed`

Рекомендуемый набор полей:

- `process_id` - идентификатор процесса;
- `__time` - timestamp сообщения;
- `var_category` - категория переменной (например, `epkData`, `startAttributes`);
- `var_path` - нормализованный путь leaf-поля (рекомендуется без конкретного индекса);
- `var_index` - индекс элемента массива (nullable для совместимости);
- `var_type` - тип значения (`string`, `number`, `boolean`, `date`, `null`, `object`, `array`);
- `var_value` - значение leaf scalar-поля (`string/number/boolean/date/null`);
- `var_value_json` - canonical JSON для значений `object/array` в режиме `raw_object_json`;
- `path_mode` - режим обработки пути (`leaf_scalar`, `leaf_object_flatten`, `raw_object_json`).

Поведение:

- для не-массивных путей в этом datasource записи не создаются;
- для массивов создается отдельная запись на каждый разрешенный warm leaf;
- для `leaf_object_flatten` объект не хранится целиком, а раскладывается на leaf warm-поля;
- для `raw_object_json` объект/массив пишется в `var_value_json` как единое значение;
- структура ориентирована на фильтрацию и агрегации по значениям и позициям элементов.

---

## 6. Правила типизации и извлечения

Поддерживаемые сценарии:

- массивы примитивов;
- массивы объектов;
- вложенные объекты внутри элементов массивов;
- object-поля, сохраняемые как warm leaf-поля через flatten;
- nullable значения.

Правила записи:

- путь должен быть явно разрешен warm-конфигом (deny by default);
- для `leaf_scalar` индексируются только scalar/null leaf-значения;
- для `leaf_object_flatten` выполняется рекурсивный обход object и сохраняются только leaf, совпавшие с whitelist;
- для `raw_object_json` значение object/array сохраняется как canonical JSON в `var_value_json`;
- если элемент содержит сложную структуру, но ни один leaf не соответствует whitelist-пути, запись не создается;
- `var_path` нормализуется без конкретного индекса, индекс хранится в `var_index`.

Запрещено:

- хранение элемента массива как сырого JSON для обхода whitelist-правил;
- запись object целиком без явного режима `raw_object_json` в конфиге.

---

## 7. Ограничения объема и защитные лимиты

Для контроля объема данных рекомендуется:

- сохранить общий лимит `warmVariablesLimit` (если уже используется);
- добавить лимит элементов на путь, например `warmArrayElementsLimitPerPath`;
- добавить лимит элементов на сообщение, например `warmArrayElementsLimitPerMessage`.
- добавить лимит leaf-полей из object на элемент, например `warmObjectLeafLimitPerElement`;
- добавить лимит глубины object, например `warmObjectDepthLimit`;
- добавить лимит размера JSON для `raw_object_json`, например `warmObjectJsonBytesLimit`.

Лимиты должны применяться после фильтрации whitelist-путей и до финальной отправки в ingestion.

---

## 8. Преимущества подхода

- Стабильная main-схема (`default_warm_main`) без разрастания колонок.
- Прозрачное и управляемое подключение массивов через warm-конфиг.
- Object-поля в массивах доступны как warm leaf-поля через `leaf_object_flatten`.
- Хранение object/array целиком в JSON возможно только точечно и явно (`raw_object_json`).
- Предсказуемая нагрузка на ingestion через лимиты.
- Удобные SQL-паттерны для поиска по массивам (`var_path`, `var_value`, `var_index`).

---

## 9. Ограничения и компромиссы

- Появляется второй datasource и необходимость запросов с JOIN в сценариях, где нужны одновременно main-данные и фильтры по массивам.
- Нужна поддержка и актуализация whitelist-конфига путей.
- Неразрешенные массивные поля не будут доступны в аналитике.
- Для `leaf_object_flatten` требуется контроль глубины/объема, иначе возможен рост числа записей.
- Для `raw_object_json` фильтрация по внутренним полям требует JSON-функций SQL-движка или дополнительного flatten.

## 10. Рекомендации по внедрению

1. Зафиксировать состав warm-полей для `default_warm_main`.
2. Зафиксировать whitelist array-путей по категориям (`tier2WarmCategories`) и режим для каждого пути (`leaf_scalar`, `leaf_object_flatten`, `raw_object_json`).
3. Определить лимиты для массивов и object-полей (per-path, per-message, depth, leaf-per-element, json-bytes).
4. Подготовить набор эталонных SQL-проверок:
   - выборки по `var_path` + `var_value`;
   - выборки по `var_path` + `var_value_json` для `raw_object_json`;
   - агрегации по `var_index`;
   - join main + arrays_indexed по `process_id`.
5. Провести сравнение с текущим вариантом по метрикам:
   - ingest latency;
   - число записей на сообщение;
   - покрытие аналитических сценариев.

---

## 11. Краткий итог

Целевой дизайн - это модель `default_warm` для основного хранения и `compcom`-подобный индексный подход для массивов, где:

- main остается компактным и фиксированным;
- массивы индексируются только по явно разрешенным warm-путям;
- object-поля в массивах могут храниться как warm leaf-поля через `leaf_object_flatten`;
- хранение object/array целиком в JSON допускается только как явное исключение (`raw_object_json`) по whitelist.
