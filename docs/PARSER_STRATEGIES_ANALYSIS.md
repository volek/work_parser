# Анализ стратегий парсера BPM-сообщений для Apache Druid

Отчёт составлен по данным из каталогов `logs/` и `query-results/`. Тестовый набор: **500 сообщений** (один и тот же набор для всех стратегий).

**Стратегии:** Default, Hybrid, EAV, Combined, Compcom (compact combined). Для **Combined** и **Compcom** поддерживается опция `parser.warmVariablesLimit` (10..1010, шаг 100): число записей в `process_variables_indexed` на одно сообщение может быть ограничено.

---

| `query-results/*.txt` | Результаты выполнения SQL-запросов к Druid по каждой стратегии (Router/Broker), статус и число строк |

## 3. Детальное сравнение по параметрам

### 3.1 Производительность парсинга и трансформа

| Параметр | Default | Hybrid | EAV | Combined | Compcom |
|----------|---------|--------|-----|----------|---------|
| Время на сообщение (parse only) | ~6,7 мс | **~6,5 мс** | ~6,1 мс | ~10,0 мс | ~7,8 мс |
| Время трансформа на сообщение | ~6,9 мс | **~1,2 мс** | ~7,9 мс | ~6,8 мс | ~8,5 мс |
| Сложность по данным | Одна wide-запись, все переменные как колонки | Одна запись, часть в JSON-блобе | Полный flatten: событие + все листовые переменные | Главная запись + warm-переменные по категориям | Как Combined без cold blob |
| Оценка памяти | Высокая (wide) | Низкая | Очень высокая (~367 записей/сообщение) | Средняя (~70 записей/сообщение) | Средняя (~70) |

**Вывод:** Hybrid лидирует по скорости трансформа и предсказуемости объёма; EAV — самый тяжёлый по объёму записей; Default — один datasource и одна запись на сообщение, но долгий трансформ из-за wide-таблицы; Combined и Compcom — компромисс по объёму.

### 3.2 Нагрузка на инфраструктуру (ingest)

| Параметр | Default | Hybrid | EAV | Combined | Compcom |
|----------|---------|--------|-----|----------|---------|
| HTTP-запросов к Overlord (submit task) | 1 | **1** | 184 | 36 | 36 |
| Объём данных на сеть | 500 широких строк | 500 широких строк | 500 + 182 804 строк | 500 + 34 614 строк | 500 + 34 614 строк |
| Время до завершения submit | ~10,1 с | **~6,2 с** | ~33,2 с | ~12,6 с | ~11,3 с |
| Зависимость от числа переменных | Рост числа колонок | Рост размера строки (блоб) | Сильный рост числа строк | Рост по «тёплым» категориям | Как Combined |

**Вывод:** Hybrid даёт минимальную нагрузку на Overlord и самое короткое время submit; EAV — максимальную; Default — один батч, но дольше Hybrid из-за размера payload (wide).

### 3.3 Модель данных и запросы

| Параметр | Default | Hybrid | EAV | Combined | Compcom |
|----------|---------|--------|-----|----------|---------|
| Число datasource | **1** | **1** | 2 | 2 | 2 |
| JOIN для типичных сценариев | Не нужен | Не нужен | Нужен (events + variables) | Нужен (main + variables_indexed) | Нужен (main_compact + variables_indexed) |
| Поиск по «горячим» полям | Прямые колонки | Прямые колонки | Прямые в `process_events` | Прямые в `process_main` | Прямые в `process_main_compact` |
| Поиск по переменным по имени/пути | Колонки `variables.<path>` | JSON по блобам (JSON_VALUE, JSON_QUERY) | Отдельные строки (`var_path`, `var_value`, `var_type`) | Индексированные «тёплые» колонки | Как Combined |
| Cold blob в Druid | Нет (всё в колонках) | Один блоб в строке | Нет | `var_blob_json` в main | **Нет** |

**Вывод:** Default и Hybrid позволяют обходиться без JOIN; у EAV, Combined и Compcom типичны запросы с JOIN для доступа к переменным. У Compcom нет cold blob — экономия места, но нет доступа к холодным переменным в Druid.

### 3.4 Гибкость и эксплуатация

| Параметр | Default | Hybrid | EAV | Combined | Compcom |
|----------|---------|--------|-----|----------|---------|
| Добавление нового типа запроса по переменным | Новые колонки при новом пути | Зависит от JSON-пути в блобе | Гибко по `var_path`/`var_value` | Гибко по «тёплым» категориям | Как Combined |
| Мониторинг и отказоустойчивость | Один DS — проще | Один DS — проще | Два DS — согласованность загрузки | Два DS — согласованность | Два DS — согласованность |
| Масштабирование при росте переменных | Рост числа колонок (очень wide) | Рост размера строки (блоб) | Сильный рост числа строк и батчей | Рост по выбранным категориям | Как Combined |

## 5. Рекомендации по целесообразности применения

### 5.1 Default

**Целесообразно применять, когда:**

- Нужна максимально простая модель: один datasource, все переменные как колонки `variables.<path>`.
- Запросы в основном по известному набору полей и переменных, без частых изменений схемы.
- Приоритет — простота мэппинга, а не минимальное время ingest.

**Ограничения:**

- Очень широкая таблица при большом разнообразии переменных; рост числа колонок; время трансформа и размер payload выше, чем у Hybrid; время submit задач больше, чем у Hybrid.

**Вердикт:** Имеет смысл для прототипов и сценариев с небольшим, стабильным набором переменных. Для минимальной нагрузки на ingest и лучшей предсказуемости схемы предпочтительнее **Hybrid**.

---

### 5.2 Hybrid

**Целесообразно применять, когда:**

- Приоритет — минимальное время парсинга, трансформа и загрузки в Druid.
- Основные сценарии — фильтрация и аналитика по «горячим» полям (process_id, process_name, state, module_id, даты и т.д.).
- Доступ к переменным нужен эпизодически и допустим через JSON-функции по блобам.
- Важна простота эксплуатации: один datasource, один тип задач индексации.

**Ограничения:**

- Поиск по произвольному атрибуту переменных — только через разбор JSON в запросе (может быть менее удобно и в части сценариев медленнее, чем индексированные колонки).

**Вердикт:** **Рекомендуется как стратегия по умолчанию** для типичных сценариев мониторинга и отчётности по процессам без жёстких требований к произвольному поиску по переменным как по реляционным атрибутам.

---

### 5.3 EAV

**Целесообразно применять, когда:**

- Нужен полноценный поиск и аналитика по переменным как по атрибутам (var_path, var_value, типы).
- Допустимы большие время парсинга/транформа и загрузки и два datasource.
- Запросы с JOIN между событиями и переменными приемлемы по производительности и сложности написания.

**Ограничения:**

- Максимальные объём записей (183 304 на 500 сообщений) и число батчей (184); самое долгое время submit (~33 с).

**Вердикт:** Имеет смысл для сценариев с акцентом на аналитику по переменным и готовностью мириться с высокой нагрузкой на парсинг и ingest. Не выбирать по умолчанию, если важны скорость загрузки и простота (один DS).

---

### 5.4 Combined

**Целесообразно применять, когда:**

- Нужен баланс: быстрые запросы по «главной» сущности (process_main) и индексированный доступ по ограниченному набору «тёплых» переменных (категорий).
- Нужен доступ к холодным переменным в Druid через `var_blob_json`.
- Приемлемы два datasource и настройка списка «тёплых» категорий.

**Ограничения:**

- Два прохода (main + по категориям) и два datasource — сложнее, чем Hybrid; при росте числа категорий растёт объём данных и число батчей; время parse максимальное среди стратегий в текущем прогоне.

**Вердикт:** Разумный компромисс между Hybrid и EAV по объёму данных и гибкости запросов по переменным, когда нужен именно индексированный доступ по части переменных и cold blob в Druid.

---

### 5.5 Compcom (compact combined)

**Целесообразно применять, когда:**

- Нужна та же модель, что у Combined (main + indexed variables), но **без хранения cold blob в Druid** (экономия места).
- Запросы по холодным переменным через `var_blob_json` не требуются; достаточно warm-переменных в `process_variables_indexed`.

**Ограничения:**

- Нет колонки `var_blob_json` в основной таблице — доступ к холодным переменным в Druid невозможен.

**Параметр `parser.warmVariablesLimit` (10..1010, шаг 100):** для Combined и Compcom ограничивает число записей в `process_variables_indexed` на одно сообщение (для тестов и вариативности объёма warm-данных).

**Вердикт:** Оптимальна для сценариев, где cold blob в Druid не нужен, при сохранении той же структуры запросов, что у Combined (по `process_main_compact` и `process_variables_indexed`). В текущем прогоне — чуть быстрее Combined по времени ingest (11,3 с против 12,6 с).

## 6. Рекомендации по изменению стратегий и применимости

### 6.1 Возможные изменения стратегий

1. **Default**
   - Рассмотреть ограничение числа колонок (например, только tier1 переменных) или перенос части переменных в один JSON-столбец, чтобы снизить ширину таблицы и время трансформа/ingest.
   - Добавить опцию «default_compact»: одна запись на сообщение с фиксированным набором колонок + один JSON для остальных переменных (аналогично Hybrid по форме, но с другим набором полей).

2. **Hybrid**
   - Расширять список hot/structured полей по мере появления частых запросов, чтобы реже обращаться к JSON.
   - Документировать рекомендуемые JSON-пути и индексы (если Druid поддерживает) для ускорения запросов по блобам.

3. **EAV**
   - Для снижения числа батчей: увеличить `batch_size_config` (если допустимо по памяти и лимитам Overlord) или агрегировать отправку (несколько батчей в одной HTTP-сессии, если API поддерживает).
   - Рассмотреть фильтрацию переменных по категориям (аналог warm) для уменьшения объёма записей при сохранении EAV-модели.

4. **Combined / Compcom**
   - Проводить A/B по разным значениям `parser.warmVariablesLimit` (10, 110, … 1010) и замерять время parse/ingest и качество запросов.
   - Явно документировать, какие категории входят в warm, чтобы упростить выбор между Combined и Compcom (нужен ли cold blob).

### 6.2 Применимость по сценариям (кратко)

| Сценарий | Предпочтительная стратегия |
|----------|----------------------------|
| Мониторинг процессов, дашборды по state/module_id, минимум задержки загрузки | **Hybrid** |
| Простейшая схема «всё в колонках», мало переменных | Default |
| Глубокая аналитика по переменным (path/value/type), ad-hoc запросы | EAV |
| Баланс: главная сущность + индексированные warm-переменные + cold в Druid | Combined |
| То же, но без cold blob в Druid (экономия места) | Compcom |

## 7. Действия для дополнительных измерений стратегий

### 7.1 Уже доступные метрики в логах

- **Parse:** `parse_ns_total`, `parse_ms_total`, `transform_ns`, `transform_ms`, `records_count`, `throughput_mb_s`, `avg_per_message_ms`, `avg_transform_per_message_ms`.
- **Ingest:** по каждому datasource — `records_total`, `batches_total`, `ingest_total_ms`, `sum_http_round_trip_ms`, `sum_spec_build_ms`; по этапу — `ingest_stage_summary` (ingest_ms).
- **Запросы:** в query-results — статус ответа и число строк (TEMP_PERF в логах при прогоне запросов: `rows_count`, `http_round_trip_ms`, `total_ms`).

### 7.2 Рекомендуемые дополнительные измерения

1. **Время выполнения запросов (latency)**  
   - Для каждой стратегии и каждого SQL-файла сохранять в лог или отдельный отчёт: `total_ms`, `rows_count`, `sql_chars`.  
   - Сводная таблица: медиана и p95 времени ответа по типовым запросам (например, select top 100, filter by epkId, join main+indexed).

2. **Масштабирование по объёму**  
   - Повторить прогоны на наборах 1K, 10K, 50K сообщений (при наличии ресурсов).  
   - Снять метрики: parse_ms_total, transform_ms, ingest_stage_summary, число батчей, число записей.  
   - Оценить линейность роста и выявить стратегии, наиболее чувствительные к росту объёма.

3. **Вариация warmVariablesLimit (Combined / Compcom)**  
   - Прогоны с `parser.warmVariablesLimit` = 10, 110, 210, … 1010 на одном и том же наборе сообщений.  
   - Замеры: records_count (variables_indexed), parse_ms_total, transform_ms, batches_total (variables_indexed), ingest_total_ms по DS variables_indexed.

4. **Размер payload и использование хранилища Druid**  
   - После успешного завершения задач индексации — запрос к Druid (если доступен API/метрики) на размер сегментов по datasource.  
   - Сравнение: Default (wide) vs Hybrid (блоб) vs EAV (много строк) vs Combined/Compcom (main + indexed) в пересчёте на одно сообщение.

5. **Успешность и стабильность запросов**  
   - Автоматический прогон всех SQL по каждой стратегии с фиксацией: HTTP status, наличие ошибки в теле ответа, число строк.  
   - Регулярное (например, при изменении схем или запросов) обновление сводки в данном отчёте (раздел 2.3 и при необходимости раздел 4).

6. **Повторяемость (несколько прогонов)**  
   - Несколько последовательных прогонов parse+ingest на одном наборе сообщений для одной и той же стратегии.  
   - Оценка дисперсии parse_ms_total и ingest_stage_summary для учёта влияния JVM/сети.

### 7.3 Формат отчёта для дополнительных измерений

Рекомендуется вести отдельный файл (например, `docs/STRATEGY_BENCHMARKS.md` или таблицу в данном документе) с колонками:

- Дата прогона, версия приложения, число сообщений.
- Стратегия, параметры (например, warmVariablesLimit).
- Метрики: parse_ms_total, transform_ms, records_count, batches_total, ingest_stage_summary_ms.
- При наличии: размер сегментов по DS, медиана/p95 времени запросов по группам запросов.

Это позволит отслеживать регрессии и эффект от изменений стратегий или конфигурации.

## 8. Итоговые выводы

1. **Hybrid** — лучший выбор по скорости парсинга, трансформа и загрузки в Druid и по простоте (один datasource, один батч). Все проверенные запросы в текущем прогоне выполняются успешно. Ограничение — поиск по произвольным переменным только через JSON по блобам.

2. **Default** — один datasource и одна запись на сообщение, но широкая таблица и большее время трансформа и submit по сравнению с Hybrid; целесообразен при простой схеме и небольшом наборе переменных.

3. **EAV** — максимальная гибкость модели (события + переменные по атрибутам), но самые большие затраты по объёму записей (183 304 на 500 сообщений) и по времени ingest (184 батча, ~33 с). Подходит для сценариев с акцентом на аналитику по переменным.

4. **Combined** — компромисс по объёму записей и времени между Hybrid и EAV; индексированный доступ по «тёплым» переменным и cold blob в Druid; два datasource, 36 батчей, ~12,6 с submit.

5. **Compcom** — как Combined, но без cold blob в Druid (`process_main_compact`); тот же лимит warm-переменных; экономия места; в текущем прогоне чуть быстрее Combined по времени submit (11,3 с).

6. В **текущем прогоне** все запросы по всем стратегиям завершились успешно (100% успешных выполнений в query-results). Для актуального сравнения при изменении схем или набора запросов рекомендуется повторить загрузку, прогон запросов и при необходимости обновить данный отчёт и выполнить действия из раздела 7.
