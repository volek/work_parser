# Анализ стратегий парсера BPM-сообщений для Apache Druid

Отчёт составлен по данным из каталогов `logs/` и `query-results/`. Тестовый набор: **100 сообщений** (один и тот же набор для всех стратегий).

## 1. Источники данных и методология

### 1.1 Генерация тестовых данных

| Параметр | Значение |
|----------|----------|

| Количество сообщений | 100 |

### 1.2 Источники метрик

| Источник | Содержание |
|----------|------------|
| `logs/*_parse.log` | Только парсинг: время, число записей (без ingest) |
| `logs/*_ingest.log` | Парсинг + подготовка записей + отправка задач в Druid (Overlord), батчи, datasource |
| `query-results/*.txt` | Результаты выполнения SQL-запросов к Druid по каждой стратегии (Router/Broker) |

## 3. Детальное сравнение по параметрам

### 3.1 Производительность парсинга

| Параметр | Hybrid | EAV | Combined |
|----------|--------|-----|----------|
| Время на сообщение (parse only) | **~3,2 мс** | ~9,1 мс | ~12,8 мс |
| Время на сообщение (parse + подготовка) | **~5 мс** | ~12,2 мс | ~16,7 мс |
| Сложность по данным | Одна запись/сообщение, часть в JSON-блобе | Полный flatten: событие + все листовые переменные | Главная запись + только «тёплые» переменные по категориям |
| Оценка памяти | Минимальная | Высокая (~161 запись/сообщение) | Средняя (~70 записей/сообщение) |

**Вывод:** Hybrid лидирует по скорости и предсказуемости; EAV — самый тяжёлый по объёму и времени; Combined — компромисс.

### 3.2 Нагрузка на инфраструктуру (ingest)

| Параметр | Hybrid | EAV | Combined |
|----------|--------|-----|----------|
| HTTP-запросов к Overlord (submit task) | **1** | 18 | 8 |
| Объём данных на сеть | Минимальный (100 широких строк) | Максимальный (16 145 строк) | Средний (6 974 строки) |
| Время до завершения submit | **~2,9 с** | ~6,5 с | ~5,1 с |
| Зависимость от числа переменных | Рост размера строки (блоб) | Сильный рост числа строк | Рост только по «тёплым» категориям |

**Вывод:** Hybrid даёт минимальную нагрузку на Overlord и сеть; EAV — максимальную; Combined — среднюю.

### 3.3 Модель данных и запросы

| Параметр | Hybrid | EAV | Combined |
|----------|--------|-----|----------|
| Число datasource | **1** | 2 | 2 |
| JOIN для типичных сценариев | Не нужен | Нужен (events + variables) | Нужен (main + variables_indexed) |
| Поиск по «горячим» полям (process_id, state, module_id и т.д.) | Прямые колонки | Прямые в `process_events` | Прямые в `process_main` |
| Поиск по переменным по имени/пути | Через JSON по блобам (JSON_VALUE, JSON_QUERY и т.п.) | Отдельные строки (`var_path`, `var_value`, `var_type`) | Индексированные «тёплые» колонки (`var_category`, `var_path`, `var_value`) |
| Типичные запросы | Простые SELECT/WHERE по колонкам + при необходимости JSON-функции | JOIN events ↔ variables, фильтры по `var_path`/`var_value` | SELECT по main, JOIN к indexed по категориям, фильтры по путям в indexed |

**Вывод:** У Hybrid все проверенные запросы выполняются без JOIN; у EAV и Combined типичны запросы с JOIN для доступа к переменным.

### 3.4 Гибкость и эксплуатация

| Параметр | Hybrid | EAV | Combined |
|----------|--------|-----|----------|
| Добавление нового типа запроса по переменным | Зависит от наличия колонки или JSON-пути в блобе | Гибко по `var_path`/`var_value` | Гибко по «тёплым» категориям и путям |
| Мониторинг и отказоустойчивость | Один datasource — проще | Два DS — нужна согласованность загрузки | Два DS — согласованность загрузки |
| Масштабирование при росте числа переменных | Рост размера строки (блоб) | Сильный рост числа строк и батчей | Рост только по выбранным категориям и батчам indexed |

## 5. Рекомендации по целесообразности применения

### 5.1 Hybrid

**Целесообразно применять, когда:**

- Приоритет — минимальное время парсинга и загрузки в Druid.
- Основные сценарии — фильтрация и аналитика по «горячим» полям (process_id, process_name, state, module_id, даты и т.д.).
- Доступ к переменным нужен эпизодически и допустим через JSON-функции по блобам.
- Важна простота эксплуатации: один datasource, один тип задач индексации.

**Ограничения:**

- Поиск по произвольному атрибуту переменных — только через разбор JSON в запросе (может быть менее удобно и в части сценариев медленнее, чем индексированные колонки).

**Вердикт:** **Рекомендуется как стратегия по умолчанию** для типичных сценариев мониторинга и отчётности по процессам без жёстких требований к произвольному поиску по переменным как по реляционным атрибутам.

---

### 5.2 EAV

**Целесообразно применять, когда:**

- Нужен полноценный поиск и аналитика по переменным как по атрибутам (var_path, var_value, типы).
- Допустимы большие время парсинга и загрузки и два datasource.
- Запросы с JOIN между событиями и переменными приемлемы по производительности и сложности написания.

**Ограничения:**

- Максимальные парсинг и ingest по времени и объёму данных; больше всего батчей и записей.

**Вердикт:** **Имеет смысл** для сценариев с акцентом на аналитику по переменным и готовностью мириться с высокой нагрузкой на парсинг и ingest. Не выбирать по умолчанию, если важны скорость загрузки и простота (один DS).

---

### 5.3 Combined

**Целесообразно применять, когда:**

- Нужен баланс: быстрые запросы по «главной» сущности (process_main) и индексированный доступ по ограниченному набору «тёплых» переменных (категорий).
- Приемлемы два datasource и настройка списка «тёплых» категорий.

**Ограничения:**

- Два прохода (main + по категориям) и два datasource — сложнее, чем Hybrid; при росте числа категорий растёт объём данных и число батчей.

**Вердикт:** **Разумный компромисс** между Hybrid и EAV по объёму данных и гибкости запросов по переменным, когда нужен именно индексированный доступ по части переменных без полного EAV.

## 6. Итоговые выводы

1. **Hybrid** — лучший выбор по скорости парсинга и загрузки в Druid и по простоте (один datasource). Все проверенные запросы в текущем прогоне выполняются успешно. Ограничение — поиск по произвольным переменным только через JSON по блобам.

2. **EAV** — максимальная гибкость модели (события + переменные по атрибутам), но самые большие затраты на парсинг и загрузку (время и объём). Подходит для сценариев с акцентом на аналитику по переменным.

3. **Combined** — компромисс по объёму записей и времени между Hybrid и EAV; даёт индексированный доступ по «тёплым» переменным при двух datasource и более сложном пайплайне, чем Hybrid.

4. В **текущем прогоне** все запросы по всем стратегиям завершились успешно (100% успешных выполнений в query-results). Для актуального сравнения при изменении схем или набора запросов рекомендуется повторить загрузку, прогон запросов и при необходимости обновить данный отчёт.
