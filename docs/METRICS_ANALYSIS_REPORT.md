## 2. Парсинг по каждой стратегии — все параметры

Парсинг выполняется по одному файлу сообщений; для каждого файла пишется операция `file_parse`, для каждого сообщения внутри файла — `parse_one`. Ниже приведены **все временные и размерные** параметры с min / max / avg по стратегиям (n — число измерений: по file_parse это файлы, по parse_one — сообщения).

**Временные (время в мс и нс):**

| Параметр | Описание |
|----------|----------|
| `read_ns` / `read_ms` | Чтение файла с диска |

**Производные:**

## 4. Select из Druid по каждой стратегии — все параметры

По каждой стратегии в `query-results/<strategy>.txt` для каждого выполненного запроса пишется `query_summary` и при необходимости `sql_request`. Ниже — **все временные и количественные** параметры: сводно по стратегии (min / max / avg по всем запросам), затем описание параметров и при необходимости детализация по запросам.

### 4.1 Описание параметров запросов

**Временные (мс и нс):**

| Параметр | Описание |
|----------|----------|
| `read_ms` | Чтение SQL-файла |
| `prepare_ms` | Подготовка запроса (подстановка параметров и т.п.) |
| `druid_exec_ms` | Время выполнения запроса в Druid (Broker + сканирование) |
| `print_ms` | Вывод результата (форматирование, консоль) |
| `query_command_total_ms` | Полное время команды запроса (read + prepare + druid_exec + print) |

**Количественные:**

| Параметр | Описание |
|----------|----------|
| `rows_count` | Число строк в ответе |
| `raw_sql_chars` | Размер исходного SQL-файла (символов) |
| `prepared_sql_chars` | Размер подготовленного SQL (символов) |
| `raw_sql_lines` / `prepared_sql_lines` | Строки в SQL |

**Производные:**

| Параметр | Описание |
|----------|----------|
| `avg_per_row_ms` | query_command_total_ms / rows_count (для интерпретации времени на строку) |

На уровне `sql_request` (ниже по стеку): `http_round_trip_ms`, `body_read_ms`, `json_decode_ms`, `total_ms`, `response_body_chars`.

- По **среднему druid_exec_ms** по стратегии: наименьшие значения у **combined** (avg 1 679 мс), затем eav (1 864 мс), hybrid (1 983 мс), default (2 264 мс), compcom (2 791 мс). Разброс по запросам большой (min 1 048–1 420 мс, max 4 360–7 161 мс).
- По **query_command_total_ms** в среднем быстрее всего **combined** (2 070 мс), затем eav (2 320 мс), hybrid (2 451 мс), default (2 871 мс), compcom (3 429 мс).

### 5.4 Применимость стратегий

- **Default:** Одна широкая таблица, простые запросы, мало батчей; подходит при небольшом числе переменных и необходимости видеть их колонками.
- **Hybrid:** Минимальное время инжеста и мало батчей; баланс между скоростью загрузки и гибкостью структуры (часть данных в JSON).
- **EAV:** Максимальная нормализация, больше всего записей и батчей, максимальное время инжеста; оправдан при сложных запросах по переменным и жёсткой схеме.
- **Combined / Compcom:** Главная таблица + индексированные переменные (warm limit 210). Compcom чуть быстрее по инжесту и без var_blob_json в основной таблице; оба варианта дают хороший баланс по времени запросов (combined в среднем быстрее по druid_exec_ms и query_command_total_ms в текущих замерах).

## 6. Воспроизведение и расширение анализа

1. **Извлечение метрик:** `python scripts/extract_metrics.py` — пишет сводную статистику в `docs/metrics_extract_output.txt`.
2. **Детализация по запросам:** из `query-results/<strategy>.txt` по строкам `operation=query_summary` извлечь `query_file`, `druid_exec_ms`, `query_command_total_ms`, `rows_count`, `read_ms`, `prepare_ms`, `print_ms` и сформировать таблицы или CSV по каждой стратегии.
3. **Повторяемость:** несколько прогонов инжеста и набора запросов для оценки дисперсии (ingest_stage_summary.ingest_ms, min/avg/max druid_exec_ms).
4. **Масштабирование:** те же замеры на 500, 2 000, 10 000 сообщений для зависимостей от объёма данных.

Текущий отчёт опирается на однократный прогон и один набор сообщений (100 файлов / 1000 сообщений); для производственных выводов целесообразно зафиксировать процедуру замеров и повторять её при изменении данных или конфигурации.
